Game idea

We are aiming to create a 2-player game that is played on one machine (pc/console) where each player takes turn fighting each other. The story line is a war between fictive and less fictive mythologies i.e. vikings, samurai, pirates, romans, egyptian gods etc. Each mythology has attacking, defending & upgrading options based on their historical or fictional characteristics. The gameplay itself is handled in different ways, first layer is a 2d “card game” on top of the 3d background environment, each player/mythology has three types of cards: offensive (red), defensive (green), upgrade (blue). They player may choose one type of card each turn and the “turn” will be based on the flipped card. An example could be playing the viking class, and choosing an offensive card, the player might then get a thunderstrike to attack the opponent with. The attack, defense and upgrade cards will change the 3d game, i.e. the thunder attack will strike the opponent in the 3d environment, likewise a defense card with a wall on it will place a wall in front of the players “castle” in the 3d environment to protect it from missiles. Each player will start with 100 HP which will decrease when attacked and regain when healing. First player who gets the opponent's HP down to zero wins the match.

Target group

Game should appeal to people of all ages and genders, we are aiming to create a strategical game, but definitely not a heavy strategical one of the kind. So it should not require a chess champion to master the game, but will benefit the player to have strategical skills. The more you play the game and get familiar with the different cards, and which cards are more likely to get than other you will get better at the game and be able to play it with strategy and beat beginners.

Platform

First version will be for PC, but the goal is to have a console ready version as well. This kind of 2-player game is most fun to play on a console where both player has a controller, a comparison could be the game “Worms” which has playable versions on pc and playstation, but is more fun on playstation if you are more than one person playing.


Goal for Unity elective “Hand-in version”

  • We want to have a playable game with 2 mythologies/classes implemented, each class must have at least one of each type of card (off, deff, upg).
  • Game must be nice looking with decent graphics; 3D models & 2D canvas.
  • Game must have a turn management system that keeps track of which player's turn it is, and how much HP each player has left etc.
  • Game must work (look good) on all screen sizes.
  • The game must be written with good code, meaning that it should run as efficiently with GPU and CPU as possible.
  • The game must contain some sound/audio effects, i.e. background music or turn based sfx.
  • We want to have a sophisticated canvas system on top of our 3D environment which looks beautiful and tweens smoothly when flipping cards etc.
  • We want the (3D) camera position/view to match each turn/card move.

Technical usage of unity features, plugins & libraries


UI canvas system

During development of “Battle of mythologies” we have been forced to explore the UI canvas system in details since half of the game is played in the 2D canvas on top of everything else. All the cards are imported png files which are then converted to unity 2D UI sprites, the sprites can then be used as objects in the canvas. The classes cards are stored in a separate canvas, i.e. the canvas holding the viking cards is called “viking cards canvas” and is only set active when the turn manager passes the turn over to the viking player. To make sure our canvas object stays the right place on the screen independent of different screen sizes we have used some different techniques. First technique is custom anchors in the “Rect Transform” component, 1 on the X axis is the entire screen width, so for this object it will start in 0.33 and end in 0.4 so it takes up 0.07 of the screen width, and 0.1 height. This technique is used for our cards.

Another technique is to access canvas objects rect transform component or position with scripts and set the anchors or positions dynamically. This technique is used for our info canvas which holds information about player name, hp etc. btw, this canvas is made with a custom bitmap font, a chapter on how we pulled this off will be found later in this document. To get a understanding of how to position UI elements dynamically here is a code snippet example:

https://github.com/rpuls/final-project/blob/master/Assets/Scripts/InfoCanvasPositionScript.cs

private void SetPositions(){

    VikingPlayerInfo.rectTransform.anchoredPosition = new Vector2 (-(Screen.width/2)+175f, (Screen.height/2)-45f);

    VikingLife.rectTransform.anchoredPosition = new Vector2 (-(Screen.width/2)+175f, (Screen.height/2)-95f);

    SamuraiPlayerInfo.rectTransform.anchoredPosition = new Vector2 ((Screen.width/2)-175f, (Screen.height/2)-45f);

    SamuraiLife.rectTransform.anchoredPosition = new Vector2 ((Screen.width/2)-175f, (Screen.height/2)-95f);

    TurnInfo.rectTransform.anchoredPosition = new Vector2 (-(Screen.width/2)+175f, -(Screen.height/2)+45f);

}

This technique is used to position UI text elements, but will this work of all elements? The short answer is no. Even though all UI elements have a rect transform component it is not necessarily possible to access it from scripts. An example is with the sliders, they do have a rect transform but you cannot manipulate it with code, instead we use the very commonly used “transform position”. Example:

https://github.com/rpuls/final-project/blob/master/Assets/Scripts/MoveScripts/ThunderTimerScript.cs

void Start() {

        m_ThunderStrikeSound = GameObject.FindGameObjectWithTag("Thunder").GetComponent<AudioSource>();

        SliderBar.transform.position = new Vector3 ((Screen.width / 2)+50, Screen.height-45 );

}

To move and manipulate our UI canvas elements we use some tweening, please jump the iTween chapter if you want to know more about how that is done.

Physics

Though our game is mostly a turn based game with, more or less, fixed animations occurring as the result of picking a card, we do have some usage of the physics offered by Unity.

Getting the cannon, or ballista depending on your team, will spawn a cannon, which will have an int increase on holding ‘x’ down, and fire upon releasing the key.

Using addForce() with the int as a parameter along with a vector3 we get from the cannon/ballista direction and afterwards a vector3 putting force upwards, we are able to fire off a cannonball gameobject - each player has one that is reused each time the cannon gets activated.

https://github.com/rpuls/final-project/blob/master/Assets/Scripts/CannonScript.cs

cannonBall.SetActive(true);

 

 

       cannonBall.transform.GetComponent<Rigidbody>().velocity = Vector3.zero;

 

       cannonBall.transform.GetComponent<Rigidbody>().angularVelocity = Vector3.zero;

 

       cannonBall.transform.position = firepos;

 

       cannonBall.transform.rotation = Quaternion.identity;

 

       cannonBall.GetComponent<Rigidbody>().AddForce(direction * force);

 

       cannonBall.GetComponent<Rigidbody>().AddForce(Vector3.up * force / 10);

The player castles are equipped with box colliders for simple collision detection, that will trigger a simple particle system giving of a cartoonish explosion, and a rigidbody for the “blow away” effect of actually hitting the opponent castle with a cannonball.

Co routines

Several scripts in our game uses a coroutine which is a special method in Unity that lets you define an event that the main thread shouldn't wait for to finish, but just execute after a number of seconds you define in the method. Here is an example:

https://github.com/rpuls/final-project/blob/master/Assets/Scripts/MoveScripts/StarScript.cs

void Start () {

    iTween.MoveTo(this.gameObject, EndPosition);

    StartCoroutine(KillYouSelf());

}

private IEnumerator KillYouSelf() {

    yield return new WaitForSeconds(animationSpeedSeconds);

    Destroy(this.gameObject);

}

In programming “yield” means that the current thread shall not be in control.

In this method we define that after a certain amount of seconds, the object should destroy itself. This can be done without the whole unity game is on pause, and will just execute after the wait time is over. Kind of like starting a separate thread. All coroutines “methods” must be declared as an IEnumerator as seen in the code example above.

Events

Events is very common thing every game has to deal with in some manner. In unity we can use the C# way of handling events through delegates. We use delegates when we want to notify a gameobject that something has happened. Here is an example of how we use delegates:

https://github.com/rpuls/final-project/blob/master/Assets/Scripts/TurnManager.cs

public class TurnManager : MonoBehaviour {

    public delegate void TurnPassed();

    public static event TurnPassed OnTurn;

        // IN a method:

OnTurn(); // tells other obejcts that turn has passed

}

public class Wall : MonoBehaviour, IMove {

TurnManager.OnTurn += TurnPassed; // Subscribes to the event.

TurnManager.OnTurn -= TurnPassed; // Unsubscribes to the event

private void TurnPassed() // Method to run when event is called

{

        if (turnCount >= turnsToStayAlive)

        {

            turnCount = 0;

            TurnManager.OnTurn -= TurnPassed;

            this.gameObject.SetActive(false);

        }

    }

}

In the first class, the Turnmanager, the event and delegate is defined. Somewhere else in the script the Turnmanager calls the OnTurn method. Then all scripts that is subscribing to the event calls their method they subscribed with, in the Wall classed that method is called TurnPassed. When the class doesn't need the event anymore it simple unsubscribes with the syntax: TurnManager.OnTurn -= TurnPassed;.

Interfaces

A big challenge we’ve had was organizing how to manage the players taking turns. As a player gets his turn he will have to draw a card with an action attached to it, some of which also needs input from the player.

After trying solutions mainly based on scripting and classes with inheritance, we finally found a solution that took advantage of Unity’s component based approach, using an interface - IMove.

Each player now has a canvas with a gameobject for each card class (Attack, Defence and Upgrade) containing a list of gameobjects with possible cards to draw.

These game objects represent the individual cards, with a corresponding action and thus have our interface script attached to them, which includes a reference.

https://github.com/rpuls/final-project/blob/master/Assets/Scripts/DoMove.cs

using System.Collections;

using System.Collections.Generic;

using UnityEngine;

public class DoMove : MonoBehaviour {

    public GameObject CardGameObj;

    internal void Activate()  {

        if (CardGameObj == null) return;

        CardGameObj.GetComponent<IMove>().DoMove();

        gameObject.SetActive(false);

    }

}

This way when a card is drawn, we can call the DoMove() method through the DoMove script and have the card perform whichever action it needs.

Afterwards to signal the end of the turn the card will invoke its cleanup and call EndTurn() in our TurnManager, if needed with an object reference if a gameobject should be deactivated.


3D assets from Sketchup models

Even though it was stated from the beginning that this project won’t be graded based on the graphical models we have decided to invest some time in 3D modelling to reach our own needs for a beautiful looking game. A game can be great with almost no graphics at all if the gameplay doesn’t require it, and is good enough alone. But if a game tries to tell a story or match a specific atmosphere, decent graphics can be absolutely crucial for the game to create the right feeling. As we want to tell a story about vikings different mythologies fighting each other, it makes no sense to use a “capsule” as player prefab, nobody would ever get the feeling that they were playing a team of vikings against samurai. The gameplay would be the same, but the whole feeling of the game would be lost. We have created several 3D models such as the two castle/house prefabs which have also been “painted” in sketchup and the exported as .dae files and material files in a material folder. Both files can be dragged into our unity project folder and then dragged into the game hierarchy. In unity the units are different than in sketchup, so imported models must be scaled down to 0.025 to match the Unity universe… Another interesting fact is that unity have switched y and z axis, in Unity the y axis is the “up” vector, but in most (maybe all) other 3D modelling programs it is the z axis which is the “up”. One workaround is to rotate the object within sketchup before exporting them, but it is also possible to rotate them within Unity.



Custom fonts

The unity UI text is a handy element to display text on the canvas, but if not using custom font it doesn’t appear very pretty. Styling text with unity is also not the best option, there is a few components you can add, such as “outline” and “shadow”, but both of them doesn’t look very nice, and if you have ever used photoshop “blending options” you know how these styles are supposed to appear. After doing some research we realises that custom bitmap font can be made for unity, so we gave that a shot to tune up the overall appearance of our game.

Using photoshop to create a png image with all a-z A-Z 0-9 etc. Style it exactly as you want - no limits. Run “shoebox” and import your png image with all the styled chars, shoebox will then analyse the image and generate a xml file that describes what part of the png image corresponds to a certain char, just like a sprite sheet. A new png is then generated with the xml file and can now be used in unity with the “Bitmap Font Importer” plugin from the asset store.

iTween

iTween is an external plugin/library that offers a class/object “iTween” which ships with some very handy tweening methods. Tweening is used in the game develop industry, but also in photo manipulation with photoshop, 3d animation softwares and so on and on. Tweening can be described as calculating an array of positions in between 2 or more given positions. With mathematical formulas we can describe the behaviour of the position tweening. Tweening can also be a lot more advanced than just calculating positions, i.e. in movie making there is sometime need for generating pictures “between” two pictures, so 2 given frames will generate “in-between-pictures” and there for the name “tweening”. In our game we are using iTween generously, both when we are moving canvas object such as flipping cards, but also when shooting throwing stars, katana, sea snake etc, even some of our cameras have tweening between one viewpoint to another, which creates a smooth and neat gaming experience! And now an example:

https://github.com/rpuls/final-project/blob/master/Assets/Scripts/MoveScripts/SeaDragonScript.cs

private Hashtable Attack = iTween.Hash("x", -97f, "y", 7f, "z", 6.5f, "time", 1);

private IEnumerator SeaAttack(){

        CameraScript.ChangeCamera (CameraView.Center);

        yield return new WaitForSeconds (2f);

        iTween.MoveTo (SeaSnakePrefab, Engange);

        yield return new WaitForSeconds (1.1f);

        if (opponentWall.activeInHierarchy) {

                iTween.MoveTo (SeaSnakePrefab, AttackFail);

                yield return new WaitForSeconds (1.1f);

        } else {

                iTween.MoveTo (SeaSnakePrefab, Attack);

                turnmanager.GameManager.playerTwo.GiveDamage (20);

                yield return new WaitForSeconds (1.1f);

        }

        CleanUp();

        turnmanager.MoveIsDone(null);

}

The clever reader will notice that the method is a IEnumerator, why is that? And why do we yield return after each iTween? First of all it is important to understand that iTween works a bit like a seperate thread, so when a iTween.MoveTo is called and starts executing, the thread that started it will immediately jump to the next line code at start executing that and keep on running. The MoveTo method takes two arguments, first arg is the GameObject that shall be moved, second argument can be a simple vector 3 if you just want to move it to another position, but if you want a little more hands on the tweening you can use a iTween hash-table which can carry a whole lot more options. In the iTween hash table we can e.g. add “time” which tells iTween how long real time the tweening should span over, and exactly therefore we are using a coroutine so that we can yield return the same amount of time after as the tweening takes, so the next line of code will be executed after the tweening is done. Another cool feature that iTween ships with is the “easetype”, there is a number of different easetypes which can all be found here http://robertpenner.com/easing/easing_demo.html where a graph is visible and shows how the specific easetype will affect the tweening behavior of iTween. To see an example please take a look in our CardScript.cs in the git repository where we use "easetype", iTween.EaseType.easeOutQuart on the canvas cards when flipping them.
